# 一、审计结论摘要

当前插件整体完成度高、架构清晰，已经具备长期运行雏形，但仍存在几类关键问题：

- **明确 bug**：如 `stripBotMention()` 正则错误
- **权限边界问题**：admin / dm.allowFrom / 群白名单语义混淆
- **状态一致性问题**：并发写配置、runtime cache 与持久配置双轨
- **媒体边界问题**：下载无大小限制、SSRF 边界弱、临时文件清理不完整
- **SDK 语义不一致**：安全摘要、pairing、action 行为与真实逻辑部分不一致

- **0.6.3 可视为完成收尾**：按当前约定的“保守 closeout”标准，0.6.3 已达到可关闭状态。
- **唯一明确未完成且已正式延期到 0.6.4 的主项**：
  - `src/channel.ts` 中 `pairing.notifyApproval()` 的 **account scope** 问题。
- **存在 1 处文档状态漂移**：

- **其余未勾选项均属非阻塞重构项**：
  - 统一媒体下载 helper
  - 统一 config mutation helper
  - 统一文本 enrich helper
  - 这些不影响 0.6.3 收尾结论。

**结论：已落实。**

本轮确认以下事项已在代码中体现：

- `historyLimit` 已做 number/finite 判定

**判断：**`0.6.3.md` 中与 `config.ts` 相关的 P1/P2 项，可认为已完成。

**判断：**
- `CQ 参数 decode`：已完成
- `改善结构化片段拼接策略，减少文本粘连`：**建议文档改判为“已部分完成 / 非阻塞残留”**，不应继续视作 0.6.3 主阻塞项

- heartbeat 当前仍是“被动续命”模式。
  - 只有收到消息才会重置 heartbeat 计时。
  - 若 NapCat 长时间不推消息但连接实际仍正常，存在空闲链路被误重连的风险。

- `connect()` 中仍使用 `removeAllListeners()` 清理旧连接。
  - 功能上目前问题不大，但会削弱旧连接 close/error 的观测性。
  - 当前记为调试/观测性边界问题。

## `src/handler.ts`

- `fetchWithLimits()` 虽然已增加大小上限，但仍通过 `arrayBuffer()` 一次性读入响应体。
  - 在缺少或错误 `content-length` 的场景下，仍可能造成较硬的内存峰值。

- 已在最后一轮收尾补丁中修复：
  - `fileData.path` 不再统一以 `URL:` 标签包装，避免把 NapCat 本地路径误导成可访问 URL。
  - pairing 流程中，给用户和管理员的关键通知失败已增加 warn 日志，提升可观测性。

## `src/gateway.ts`

- `groupPolicy` 非法值导致群聊路径进入未定义行为的问题，已随 `config.ts` 的枚举收敛一并修复。

- 已在最后一轮收尾补丁中部分修复：
  - 被踢群、被禁言、好友新增、自动同意好友/群邀请、群 pairing 提醒等关键管理员通知路径已增加 warn 日志。
  - 剩余低价值通知若仍有静默路径，已不再构成 0.6.3 阻塞项。

## `src/outbound.ts`

- 已在收尾复查后修复：
  - 本地文件发送已增加大小限制，避免超大文件同步读盘后直接 base64 化。
  - 发送语音/视频前若文本预发送失败，现在会直接返回失败，不再继续发媒体造成“文本丢失、只剩媒体”的错误行为。

## `src/channel.ts`

- 已在最后一轮收尾补丁中修复：
  - `sendAttachment` 已增加更严格的 base64 合法性验证（不仅检查字符集，也做解码/回环校验）。

- 已在收尾复查后修复：
  - `sendAttachment / read / send / reply` 的 target 解析已统一补到更严格的 normalize/assert，避免不同 action 对非法 target 容忍度不一致。

## `src/onboarding.ts`

- 已在收尾复查后修复：
  - `configure()` 写入账号配置时已改为基于现有 account 结构合并，避免重跑向导时整对象覆盖已有字段。

- onboarding 中 `groupPolicy` 的初始值固定为 `pairing`，但若用户在已存在账号上重复执行向导，当前不会读取旧值作为初始值。
  - 这仍属于可用性/一致性问题，但不再伴随整对象覆盖风险。

## `src/config-schema.ts`

- `accounts.*.groupAllowFrom` 的 schema 仍允许 number / string 混合，但 UI hint 说明更偏向 `g群号` 字符串格式。
  - 与运行时现在对群号格式逐步收敛（尤其内存态只存纯数字）之间存在文档/约束层面的轻微漂移。

- 已在收尾复查后修复：
  - `historyLimit` 已增加 `minimum/maximum`（1~500），减少控制面板输入极端值的风险。

- `required: ["wsUrl"]` 仍然会让 disabled account 也被 schema 视为必须提供 `wsUrl`。
  - 这对“先建占位账号、暂不启用”的配置流不够友好，属于 schema 设计边界问题。
