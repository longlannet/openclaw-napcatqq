# 二、P1 必做项

## A. `src/channel.ts` — pairing approval account scope

### 目标
让 `pairing.notifyApproval()` **按请求所属 account 定向发送批准通知**，避免当前“找到第一个在线 client 就发”的错误语义。

### 当前问题
当前实现：
- 枚举 `listAccountIds(cfg)`
- 找到第一个在线 client
- 直接向 `id` 发送 `PAIRING_APPROVED_MESSAGE`

这在多账号场景下会导致：
- 批准通知可能从**错误账号**发出
- 正确账号离线时，可能被错误 fallback 掩盖
- 行为不可解释，不利于排障

### Checklist
- [ ] 为 `pairing.notifyApproval()` 设计 account-scoped 路由
- [ ] 禁止回退到“第一个在线 client”这种隐式策略
- [ ] 明确以下场景行为：
  - [ ] request 所属 account 可定位且在线
  - [ ] request 所属 account 可定位但离线
  - [ ] request 所属 account 无法定位
- [ ] 为失败路径补足 warn 日志，日志中带上 request id / target id / accountId / fallback decision
- [ ] README 中补充多账号 pairing 审批通知语义

### 设计约束
- 正确性优先于“尽量发出去”
- 若无法确认 account，**宁可 warn + 不发，也不要错账号发**

# 三、P2 应做项

## A. `src/inbound.ts` — String 模式 CQ 解析边界

### 当前复核结论
0.6.3 已经通过 `pushTextPart()` 明显改善了结构化片段拼接，但仍残留：
- String 模式 CQ 参数解析仍用 `split(",")`
- 复杂值或非严格编码下可能误切
- `textParts.join("")` + 局部手工补空格的方式仍偏脆弱

### Checklist
- [ ] 评估是否为 String 模式实现更稳的 CQ 参数解析器
- [ ] 明确哪些 segment 应强制换行、哪些应补空格、哪些应原地拼接
- [ ] 将 `textParts.join("")` 的策略进一步收敛为统一 builder/helper
- [ ] 回归检查以下片段组合：
  - [ ] 文本 + 图片
  - [ ] 文本 + 文件
  - [ ] 文本 + 转发摘要
  - [ ] 多个结构化片段连续出现
  - [ ] String 模式 CQ + 尾部纯文本

### 说明
这项属于**0.6.3 已部分完成，0.6.4 做收口**，而不是从零开始。

---

## B. `src/ws-client.ts` — heartbeat 与旧连接观测性

### 当前复核结论
主 bug 已在 0.6.3 修掉，但还留有两个值得收口的问题：
- heartbeat 目前是“**收到消息才续命**”的被动模式
- `connect()` 中对旧连接执行 `removeAllListeners()`，调试观测性偏弱

### Checklist
- [ ] 评估是否改为主动 ping/heartbeat 策略，避免长时间无消息时误判断连
- [ ] 评估是否保留旧连接部分 close/error 观测日志，而不是直接 `removeAllListeners()` 抹平
- [ ] 明确 heartbeat timeout 与 NapCat / 反代空闲超时的关系，并补简短注释

### 预期结果
- 更少“空闲链路误重连”
- 更清楚的断线诊断信息

---

## C. `src/config-schema.ts` — schema 与运行时收敛对齐

### 当前复核结论
运行时已比 schema 更保守，但 schema 侧还有两处可收口问题：
- `groupAllowFrom` schema 允许 number/string 混合，但 UI hint 更偏 `g群号`
- `required: ["wsUrl"]` 会让 disabled / 预占位账号也被 schema 视为必须配 `wsUrl`

### Checklist
- [ ] 明确 `groupAllowFrom` 的推荐表达形式，并在 schema / UI hint / README 中统一
- [ ] 评估是否放宽 disabled account 的 `wsUrl` 必填约束
- [ ] 若不改 schema 结构，也至少补注释说明这是控制面板建模权衡
